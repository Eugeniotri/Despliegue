Resumen completo explicado fácil

Idea general: un servidor de aplicaciones permite ejecutar programas en un servidor y que los usuarios accedan a ellos remotamente, normalmente desde un navegador.

1. Qué es un servidor de aplicaciones

Un servidor web sirve páginas. Un servidor de aplicaciones, además, ejecuta la lógica de negocio de la aplicación. Eso significa que no solo enseña información, sino que también procesa datos, toma decisiones y puede acceder a bases de datos.

  • Ventajas: centralización, menos administración en clientes, actualizaciones más fáciles.
  • Desventajas: más superficie de ataque, dependencia del servidor, necesidad de hardware potente.

2. Java

Java es un lenguaje y también una plataforma. No genera código máquina directo, sino bytecode, que necesita la JVM para ejecutarse.

.java → compilación → bytecode (.class) → JVM → ejecución

Gracias a esto, Java es multiplataforma: compilas una vez y puedes ejecutarlo en cualquier sistema con JVM.

3. Ediciones de Java

  • Java SE: edición básica.
  • Java ME: móviles y sistemas integrados.
  • Java EE: aplicaciones empresariales. Ahora se llama Jakarta EE.

4. Arquitectura por capas

Cliente → Presentación → Negocio → Datos
  • Cliente: navegador, app móvil o de escritorio.
  • Presentación: transforma la información a formato legible por el usuario, normalmente HTML.
  • Negocio: realiza la lógica principal de la aplicación.
  • Datos: accede a la base de datos o a servicios de directorio.

5. Jakarta EE y Apache

Jakarta EE es un conjunto de especificaciones. Tomcat implementa solo algunas y por eso no es un servidor de aplicaciones completo. TomEE amplía Tomcat y sí se considera completo.

6. Apache Tomcat

Tomcat es un contenedor de servlets libre. No es un servidor de aplicaciones completo.

  • Catalina: contenedor de servlets.
  • Jasper: convierte JSP en servlets.
  • Puerto por defecto: 8080.

7. Directorios importantes

  • bin: scripts.
  • conf: configuración. Archivo importante: server.xml.
  • lib: librerías.
  • logs: registros.
  • temp: temporales.
  • webapps: aplicaciones.
  • work: trabajo temporal.

8. Variables de Tomcat

  • CATALINA_HOME: instalación de Tomcat.
  • CATALINA_BASE: una instancia concreta.

Sirven para poder ejecutar varias instancias de Tomcat en la misma máquina.

9. Aplicaciones en Tomcat

Las aplicaciones van en webapps y pueden estar como carpeta o como archivo .war.

miApp/
 ├─ archivos HTML, JSP, CSS, JS...
 └─ WEB-INF/
    ├─ web.xml
    ├─ classes/
    └─ lib/
  • web.xml: configuración de la aplicación.
  • classes: archivos .class.
  • lib: archivos .jar.

10. Manager de Tomcat

Es el panel para gestionar aplicaciones. Permite arrancar, parar, recargar, replegar y desplegar archivos .war.

11. JSP y servlets

Servlet

Clase Java que responde a peticiones HTTP y genera respuestas dinámicas.

JSP

HTML con código Java dentro. Antes de ejecutarse, se convierte en servlet.

Servlet → procesa
JSP → muestra

12. Cookies

Las cookies se guardan en el cliente y se envían con cada petición. Sirven para recordar preferencias y personalizar contenido.

13. Sesiones

Las sesiones se guardan en el servidor y mantienen información del usuario entre peticiones. Se usan mucho para login.

Diferencia clave: cookie = cliente. sesión = servidor. La sesión es más segura.

14. Seguridad en Tomcat

La seguridad debe aplicarse en varias capas: cliente, red, servidor, Tomcat y datos. Esto se llama defensa en profundidad.

  • Ejecutar Tomcat con pocos privilegios.
  • Eliminar aplicaciones de ejemplo.
  • Ocultar la versión de Tomcat.
  • Proteger el panel de administración.
  • Usar HTTPS / TLS.

15. HTTPS, SSL/TLS y certificados

  • HTTP: no cifra.
  • HTTPS: cifra la comunicación.
  • TLS: aporta autenticación, confidencialidad e integridad.
  • El servidor necesita un certificado.
  • Un certificado firmado por una autoridad certificadora es más confiable que uno autofirmado.

Chuleta de 30 segundos

Servidor de aplicaciones = lógica de negocio.

Java = bytecode + JVM + multiplataforma.

Java EE ahora = Jakarta EE.

Capas: cliente → presentación → negocio → datos.

Tomcat = contenedor de servlets libre, no servidor completo.

Catalina = servlets. Jasper = JSP a servlet.

8080 = puerto típico de Tomcat.

webapps = aplicaciones. .war = app empaquetada.

web.xml = configuración.

Servlet = procesa. JSP = muestra.

Cookie = cliente. Sesión = servidor.

Login = sesión.

HTTPS / TLS = seguridad en tránsito.

Ideas trampa que suelen caer

  • Tomcat no es un servidor de aplicaciones completo.
  • JSP no se ejecuta directo: se convierte en servlet.
  • Recargar en Manager sirve para aplicar cambios.
  • CATALINA_HOME no es lo mismo que CATALINA_BASE.
  • Cookie y sesión no son lo mismo.
